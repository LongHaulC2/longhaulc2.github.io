Commands

This document outlines the available commands for the implant, grouped by functionality.

Command Tree​

Below are the commands and capabilities pulled directly from the Web Client help menu. See below for individual command documentation

------
System
------
exit                 : Exit the implant. This will kill the implant process on the host, don't expect a response back.
sleep                : Sleep for a specified number of seconds on the host. Ex: `sleep 5`

-----------
File System
-----------
cd                   : Change the current working directory on the host. Ex: `cd C:\Users\`
ls                   : List the contents of a directory on the host. Ex: `ls C:\Users\`
file download        : Get a file from the host the implant is running on. Ex: `file download C:\Users\user\file.txt`
file upload          : Upload a file to the host. Ex: `file upload C:\Users\user\file.txt <base64_contents>` (or use the UI)

------------
Memory Store
------------
memstore list        : List all file names in the memstore. Ex: `memstore list`
memstore upload      : Upload a file to the implant memstore. Ex: `memstore upload <file_name> <base64_contents>`
memstore download    : Download a file from the implant memory store. Ex: `memstore download <file_name>`
memstore delete      : Delete a file from the implant's memory store. Ex: `memstore delete <file_name>`
memstore clear       : Clear *all* files in the implant's memory store. Ex: `memstore clear`

-----------
C2 Strategy
-----------
strat active         : List the active strategy for the implant.
strat list           : List the available strategies for the implant.
strat post           : Set the post strategy for the implant. Ex: `strat post my_post_strategy`
strat get            : Set the get strategy for the implant. Ex: `strat get my_get_strategy`

---------
Execution
---------
bof                  : Run a BOF. Ex: `bof <base64_bof_object> <args>` OR from memstore: `bof *memstore_bof_name <args>`

A Note on Command Outputs​

Traditional Command and Control (C2) frameworks often mix status messages, error codes, and command output into a single text blob. We take a more structured approach to make parsing and automation easier.

Every response from the implant is broken down into at least three distinct fields:

data: The raw payload returned by the command. This is the actual "answer" to your task (e.g., a file listing, process list, or file contents).

message: A human-readable status message. This is typically a translation of the Windows API response, for direct transparency as to what is happening on the system, or a custom status message for commands where Windows Errors don't fit.

  • Ex: ERROR_SUCCESS (0) becomes "Success", via FormatMessage with the FORMAT_MESSAGE_FROM_SYSTEM flag

  • Ex: 5 becomes Access Denied

  • See FormatMessage function (winbase.h)

windows_error_code: The specific Windows API error code or internal return value. Useful for debugging or programmatic handling of errors.

  • System Error Codes (0-499)

Additional fields may be included as well, for tasks where there are multiple outputs:

Example:

Strat active returns a comms_get_strategy, and a comms_post_strategy. These variations will be noted in the documentation below

019c62bb > strat active
--- comms_get_strategy ---
http_get_10_0_0_30_9090_aabb
--- comms_post_strategy ---
http_post_10_0_0_30_9090_aabb

Technical/API response Structure:​

For reference - this is the structure of the task response, with the values listed above. If you aren't using the API, don't worry about this. More details can be found in the API docs (link those when done...)

{
  "task_uuid": "00000000-0000-0000-0000-000000000000",
  "implant_uuid": "00000000-0000-0000-0000-000000000000",
  "result": {
    "message": {
      "type": "text",
      "value": "Success"
    },
    "data": {
      "type": "text",       # or bytes, if content contains binary data.
      "value": "some_data" 
    }
    "windows_error_code": {
      "type": "text",
      "value": "0"
    }
  }
}

0. Special Characters​

A collection of special characters, and what they mean.

CommandDescription
*"de-ref", used to access a file from the memory store in the command

The Dereference Operator (*)​

The dereference operator (*) lets you use binary data from the implant's memory store system as a command argument, to any input that takes binary (or base64) data.

WARNING: BINARY DATA ONLY: The dereference operator is designed exclusively for handling raw binary blobs (e.g., executables, shellcode). This makes it great for commands that take raw data, but not suitable for text data. The system treats referenced data as raw bytes; attempting to use text will result in illegible garbage output, or unexpected crashes.

Note, this syntax is only currently supported on the file upload, and bof commands.

How is this useful

The deref operator has a few advantages, mainly by allowing you to transfer a tool or payload once and deploy it multiple times from memory, rather than re-sending the same data over the network for every operation. Granted, there is a tradeoff, increased memory useage on the implant to store the data.

Syntax

<command> *<arg>

Example

File Upload:

A traditional file upload:

file upload C:\Temp\not_malware.exe <base64_blob>

Dereferenced Upload:

If my_payload is already in the memory store, this will pull that data, and write to the said file.

file upload C:\Temp\not_malware.exe *my_payload

1. System Commands​

Control the state and execution of the implant process itself.

CommandDescription
exitTerminate the implant process on the host.
sleepSleep for a specified number of seconds on the host.

exit​

Exits the implant execution.

This kills the implant process on the host. Do not expect a response confirmation from the implant after this command is issued.

  • Usage: exit

Return Values:

Does not return.

sleep​

Updates the implant sleep value, to a specified number of seconds.

  • Usage: sleep <seconds>
  • Example:
sleep 5

Return Values

data: Sleep time that the implant was updated to

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

2. File System Operations​

Interact with files on the host's disk.

CommandDescription
cdChange the current working directory on the host.
lsList the contents of a directory on the host.
file downloadDownload a file from the host machine to the server.
file uploadUpload a file from the server to the host machine.

cd​

Changes the current working directory on the host machine.

  • Usage: cd <directory>
  • Example:
cd C:\Users\

Return Values

data: "" (Nothing, empty string)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

ls​

Lists the contents of a directory on the host. If no directory is provided, it defaults to the current working directory (.).

  • Usage: ls [directory]
  • Example:
ls C:\Users\

Return Values

data: A newline (\n) separated list of files and directories in the listed directory. If using the api, split by \n.

  • Ex: ls_output = data.split(\n)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

file download​

Retrieves a file from the host machine where the implant is running.

  • Usage: file download <file_path>
  • Example:
file download C:\Users\user\file.txt

Return Values

data: The file contents of the file retrieved from the system. Via the GUI, this is printed to the screen as base64.

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

file upload​

Uploads a file to the host machine.

Note: When using the GUI, input file contents as a Base64 string. The interface automatically decodes this to binary before serializing the task into MessagePack. If you are using the API directly, you can construct the MessagePack task using raw binary.

  • Usage: file upload <file_path> <base64_content>
  • Example:
file upload C:\Users\user\file.txt <base64_file_contents>

Return Values

data: "" (Nothing, empty string)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

3. C2 Strategy​

Manage how the implant communicates with the C2 server.

CommandDescription
strat postSet the "post" strategy (data exfiltration method).
strat getSet the "get" strategy (task retrieval method).
strat listList all available strategies on the implant.
strat activeList the currently active strategies.

strat post​

Sets the posting strategy (how the implant sends data back).

  • Usage: strat post <strategy_name>
  • Example:
strat post my_post_strategy

strat get​

Sets the getting strategy (how the implant retrieves tasks).

  • Usage: strat get <strategy_name>
  • Example:
strat get my_get_strategy

strat list​

Lists the available strategies configured on the implant.

  • Usage: strat list

strat active​

Displays the currently active strategies.

  • Usage: strat active

Return Values

data: "" (Nothing, empty string)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

Custom fields:

This command returns specific key-value pairs in the response object. This structure is preferred over newline-delimited strings in the data field, as it allows the UI and API to parse active strategies reliably without ambiguity.

comms_get_strategy: The name of the active Ingress strategy (the method the implant uses to retrieve tasks).

comms_post_strategy: The name of the active Egress strategy (the method the implant uses to exfiltrate data).

Ex:

019c62bb > strat active
--- comms_get_strategy ---
http_get_10_0_0_30_9090_aabb
--- comms_post_strategy ---
http_post_10_0_0_30_9090_aabb

4. Memory Store​

Interact with the implant's volatile memory storage (RAM-only file system). See MemoryStore docs for additional implementation info.

CommandDescription
memstore listList all filenames currently in the memory store.
memstore uploadUpload a file into the implant's internal memory store.
memstore downloadDownload a file from the implant's internal memory store.
memstore deleteDelete a specific file from the memory store.
memstore clearClear all files from the memory store.

memstore list​

Lists the names of all files currently stored in the memory store.

  • Usage: memstore list

Return Values

data: All files in the memstore, by name, that are compiled into the implant. Separated via newlines (\n)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

memstore upload​

Uploads a file directly to the implant's volatile memory store (RAM). Useful for staging files without touching the disk.

Note: When using the GUI, input file contents as a Base64 string. The interface automatically decodes this to binary before serializing the task into MessagePack. If you are using the API directly, you can construct the MessagePack task using raw binary.

  • Usage: memstore upload <file_name> <base64_content>
  • Example:
memstore upload my_tool.exe aabbcc==

Return Values

data: "" (Nothing, empty string)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

memstore download​

Retrieves a file from the implant's memory store.

  • Usage: memstore download <file_name>
  • Example:
memstore download some_file_in_memstore

Return Values

data: The file contents of the file retrieved from the memstore. Via the GUI, this is printed to the screen as base64.

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

memstore delete​

Removes a specific file from the implant's memory store.

  • Usage: memstore delete <file_name>
  • Example:
memstore delete not_more_malware

Return Values

data: "" (Nothing, empty string)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

memstore clear​

Wipes all files from the memory store.

  • Usage: memstore clear

Return Values

data: "" (Nothing, empty string)

message: Windows Error Code translated via FormatMessage

windows_error_code: The resulting Windows Error Code

5. Execution​

The Host Execution commands provide operators with various mechanisms to run commands, payloads, and post-exploitation tools directly on a compromised target.

CommandDescription
run_bofLoad, initialize, and execute a compiled BOF in memory and return its standard output.

run_bof​

Executes a compiled Beacon Object File (BOF) within the implant's process space. The command takes the raw BOF object code, packs the provided arguments into the BOF's required structure, executes the entry point, and captures the resulting standard output before safely cleaning up the execution context.

Note: When using this command via the API or GUI, ensure the BOF bytes are properly passed (e.g., via base64 in the CLI or an existing memory store artifact) along with the argument string.

  • Usage: run_bof <bof_file> "<arguments>"
  • Example:
run_bof *enumerate_domain "-domain contoso.local"

Return Values

data: A string containing the captured standard output from the BOF execution. Returns empty ("") if execution failed or there was no output.

message: A descriptive error message corresponding to the resulting Windows Error Code.

windows_error_code: The status code resulting from the execution attempt. Expected codes include:

  • 0 (ERROR_SUCCESS): The BOF executed and output was successfully retrieved.
  • ERROR_INTERNAL_ERROR: The BOF launcher failed to initialize, or the BOF failed to execute.
  • ERROR_BAD_FORMAT: The provided BOF bytes could not be parsed or loaded from memory.
  • ERROR_OUTOFMEMORY: The argument structure failed to initialize.
  • ERROR_UNIDENTIFIED_ERROR: The BOF finished, but no output string could be retrieved from the context.
